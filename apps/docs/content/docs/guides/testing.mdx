Testing in fluo follows the same principle as the runtime: dependencies are explicit, module graphs are compiled deliberately, and request-pipeline behavior is verified without relying on legacy TypeScript metadata.

Use `@fluojs/testing` when you need to test providers, module wiring, guards, interceptors, DTO validation, request bodies, synthetic principals, or serialized responses.

## Install

```bash
pnpm add -D @fluojs/testing vitest
```

If you use `@fluojs/testing/vitest`, also install `@babel/core` because `fluoBabelDecoratorsPlugin()` invokes Babel at runtime.

```bash
pnpm add -D @babel/core
```

## The fluo testing ladder

Build tests from the smallest explicit dependency boundary outward.

| Level | File location | Use for | Main API |
| --- | --- | --- | --- |
| Unit | `src/**/*.test.ts` | Direct class behavior and small failure branches | direct construction, `@fluojs/testing/mock` |
| Slice/module | `src/**/*.slice.test.ts` | DI wiring, provider visibility, overrides | `createTestingModule({ rootModule })` |
| HTTP e2e-style | `test/**/*.e2e.test.ts` | Routes, guards, interceptors, DTOs, serialization | `createTestApp({ rootModule })` |
| Platform/conformance | package tests | Adapter/runtime package contracts | conformance subpaths |

```txt
src/users/
  users.service.test.ts
  users.controller.test.ts
  users.slice.test.ts

test/
  app.e2e.test.ts
```

fluo differs from NestJS by requiring tests to name an explicit `rootModule`. The testing utilities compile the module graph you authored instead of inferring dependencies from reflection metadata.

## Request-level tests

Use `createTestApp({ rootModule })` as the default app-developer path for HTTP behavior.

```ts
import { createTestApp } from '@fluojs/testing';

const app = await createTestApp({ rootModule: AppModule });

const response = await app
  .request('POST', '/users/')
  .header('x-request-id', 'test-request-1')
  .query('include', 'profile')
  .principal({ subject: 'user-1', roles: ['admin'] })
  .body({ name: 'Ada' })
  .send();

expect(response.status).toBe(201);

await app.close();
```

`app.request(...).send()` keeps tests close to HTTP semantics without requiring manual `FrameworkRequest` or `FrameworkResponse` stubs. Use it for route behavior, guards, interceptors, DTO validation, query parameters, headers, request bodies, synthetic principals, and response serialization.

Keep lower-level helpers such as `app.dispatch(...)`, `makeRequest(...)`, and raw runtime bootstrap tests for adapter/runtime contracts where the dispatch boundary itself is what the test must prove.

`createTestApp(...)` accepts the same bootstrap options as the runtime HTTP bootstrap, including providers, filters, converters, interceptors, middleware, observers, versioning, and diagnostics options. The testing helper prepends its request-context middleware while preserving caller-provided middleware in the same app middleware chain.

## Module and slice tests

Use `createTestingModule(...)` when the contract is DI wiring, provider visibility, or replacement of cross-cutting behavior.

```ts
import { createTestingModule } from '@fluojs/testing';
import { vi } from 'vitest';

const module = await createTestingModule({ rootModule: AppModule })
  .overrideProvider(USER_REPOSITORY, {
    create: vi.fn().mockResolvedValue({ id: '1', name: 'Alice' }),
  })
  .compile();

const service = await module.resolve(UserService);
```

The builder also supports `overrideGuard(...)`, `overrideInterceptor(...)`, and `overrideFilter(...)` for route-pipeline tests that need to replace cross-cutting behavior before compilation.

### Preserve module identity

`overrideModule(source, replacement)` swaps imported modules while preserving the authored root module and compiled module identities. This keeps diagnostics and graph assertions tied to your application module classes instead of synthetic test-only wrappers.

```ts
const module = await createTestingModule({ rootModule: AppModule })
  .overrideModule(StripeModule, FakeStripeModule)
  .compile();

expect(module.rootModule).toBe(AppModule);
expect(module.modules.some((compiledModule) => compiledModule.type === BillingModule)).toBe(true);
```

## Mock helpers

Use the explicit mock subpath for typed mocks.

```ts
import { createDeepMock, createMock } from '@fluojs/testing/mock';
import { vi } from 'vitest';

const repo = createMock<UserRepository>({ findById: vi.fn() });
const mailer = createDeepMock(MailService);
```

`asMock(value)` narrows an existing value to a mock-friendly type, and `mockToken(token, value)` creates a provider override tuple for token-based dependencies. `createMock(..., { strict: true })` rejects access to unspecified members.

## What to test

| Feature | Recommended test |
| --- | --- |
| Service logic | Unit test with explicit fake dependencies. |
| Provider registration | Slice test with `createTestingModule({ rootModule })`. |
| Controller request/response | `createTestApp(...).request(...).send()`. |
| Guards and scopes | Request-level test with `.principal(...)` or auth headers. |
| DTO validation | Invalid and valid request bodies through the request pipeline. |
| Serialization | Assert sensitive fields are absent from response payloads. |
| Interceptors | Request-level tests, or override them in slice tests when behavior is not under test. |
| Runtime adapters | Conformance harnesses from explicit testing subpaths. |

## Conformance and portability harnesses

Application code should rarely need conformance harnesses. They are for framework-facing packages such as runtime adapters.

Use responsibility-specific subpaths:

- `@fluojs/testing/platform-conformance`
- `@fluojs/testing/http-adapter-portability`
- `@fluojs/testing/web-runtime-adapter-portability`
- `@fluojs/testing/fetch-style-websocket-conformance`

## Public API summary

| Area | API |
| --- | --- |
| Root package | `createTestingModule(...)`, `createTestApp(...)`, module introspection helpers, shared testing types |
| App subpath | `@fluojs/testing/app` |
| Module subpath | `@fluojs/testing/module` |
| HTTP helpers | `@fluojs/testing/http` |
| Mock helpers | `@fluojs/testing/mock` |
| Types | `@fluojs/testing/types` |
| Vitest tooling | `@fluojs/testing/vitest` with `fluoBabelDecoratorsPlugin()` |

## Common mistakes

| Mistake | Better approach |
| --- | --- |
| Testing route behavior by calling controller methods directly | Use `createTestApp(...).request(...).send()` so guards, interceptors, DTOs, and response writing run. |
| Building synthetic wrapper modules for every test | Compile your authored `rootModule` and override what needs replacing. |
| Mocking too much at the e2e-style level | Keep app-level tests focused on request-pipeline contracts; use unit tests for business branches. |
| Using conformance harnesses for ordinary app features | Reserve them for adapter/runtime package contracts. |
| Forgetting to close test apps | Always `await app.close()` to release lifecycle resources. |
