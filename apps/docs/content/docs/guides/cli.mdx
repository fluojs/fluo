`@fluojs/cli` is the canonical command-line entrypoint for fluo. It creates projects, generates feature slices, runs app lifecycle commands, migrates code toward standard decorators, and exports runtime inspection data.

## Install

```bash
pnpm add -g @fluojs/cli
```

Or run without installing globally:

```bash
pnpm dlx @fluojs/cli new my-app
```

Check the installed version:

```bash
fluo version
fluo --version
fluo -v
```

Interactive runs may check the public npm `latest` dist-tag and offer to update the global CLI. CI, non-TTY output, npm-script contexts, registry failures, rerun-after-update contexts, and `--no-update-check` skip that prompt.

## Create a project

The shortest path is:

```bash
fluo new my-app
cd my-app
pnpm dev
```

`fluo create` is an alias for `fluo new`.

Use `--print-plan` when you want to preview the resolved starter without creating files, installing dependencies, or initializing git.

```bash
fluo new my-app --shape application --runtime node --platform fastify --print-plan
fluo new my-service --shape microservice --transport tcp --print-plan
fluo new my-mixed-app --shape mixed --print-plan
```

## Application starters

Node.js HTTP application starters share the same install/build flow.

```bash
fluo new my-app --shape application --transport http --runtime node --platform fastify
fluo new my-express-app --shape application --transport http --runtime node --platform express
fluo new my-node-app --shape application --transport http --runtime node --platform nodejs
```

Runtime-native starters use runtime-specific entrypoints and scripts.

```bash
fluo new my-bun-app --shape application --transport http --runtime bun --platform bun
fluo new my-deno-app --shape application --transport http --runtime deno --platform deno
fluo new my-worker-app --shape application --transport http --runtime cloudflare-workers --platform cloudflare-workers
```

Generated non-Deno HTTP starters include a TDD-first Vitest layout:

- fast unit tests colocated under `src/**`,
- slice tests such as `*.slice.test.ts`,
- app dispatch tests in `src/app.test.ts`,
- default e2e-style tests in `test/app.e2e.test.ts` using `createTestApp({ rootModule })`.

## Microservice and mixed starters

Microservice starters use `--shape microservice`. TCP is the default when `--transport` is omitted.

```bash
fluo new my-microservice --shape microservice --transport tcp --runtime node --platform none
fluo new my-redis-streams-service --shape microservice --transport redis-streams --runtime node --platform none
fluo new my-nats-service --shape microservice --transport nats --runtime node --platform none
fluo new my-kafka-service --shape microservice --transport kafka --runtime node --platform none
fluo new my-rabbitmq-service --shape microservice --transport rabbitmq --runtime node --platform none
fluo new my-mqtt-service --shape microservice --transport mqtt --runtime node --platform none
fluo new my-grpc-service --shape microservice --transport grpc --runtime node --platform none
```

Supported microservice starter transports are exactly `tcp`, `redis-streams`, `nats`, `kafka`, `rabbitmq`, `mqtt`, and `grpc`.

Mixed starters generate one Fastify HTTP app with an attached TCP microservice in the same project.

```bash
fluo new my-mixed-app --shape mixed --transport tcp --runtime node --platform fastify
```

## Lifecycle commands

Generated Node.js package scripts delegate to:

```bash
fluo dev
fluo build
fluo start
```

The CLI prepends the project-local `node_modules/.bin`, defaults `NODE_ENV` to `development` for `dev` and `production` for `build`/`start`, and shows application stdout/stderr by default.

Use dry-run mode to inspect the command boundary:

```bash
fluo dev --dry-run
fluo build --dry-run
fluo start --dry-run
```

Generated Node apps use a fluo-owned restart boundary by default. The dev runner watches source/config inputs, debounces atomic saves, hashes file content before restart, loads `.env` for each child process, and ignores noisy paths such as `node_modules`, `dist`, `.git`, `.fluo`, coverage, caches, and editor swap files.

Bun, Deno, and Workers projects default to runtime-owned watch loops. Use `fluo dev --runner fluo` or `FLUO_DEV_RUNNER=fluo` if those projects should use the fluo restart runner.

Reporter flags control CLI process output, not application logging:

```bash
fluo dev --reporter pretty
fluo dev --reporter stream
fluo dev --verbose
FLUO_VERBOSE=1 fluo dev
fluo build --reporter silent
```

## Generate features

Use `fluo generate` to create feature slices and framework artifacts.

```bash
fluo generate module users
fluo generate module users --with-test
fluo generate resource users
fluo generate resource users --with-slice-test
fluo generate e2e users
fluo generate controller users
fluo generate service users
fluo generate request-dto users CreateUser
fluo generate service users --dry-run
```

Supported generator kinds and aliases:

| Kind | Aliases | Wiring behavior |
| --- | --- | --- |
| `controller` | `co` | Auto-registers |
| `service` | `s` | Auto-registers |
| `repo` / `repository` | — | Auto-registers |
| `guard` | `gu` | Auto-registers |
| `interceptor` | `in` | Auto-registers |
| `middleware` | `mi` | Auto-registers |
| `module` | `mo` | Files-only |
| `request-dto` | `req` | Files-only |
| `response-dto` | `res` | Files-only |
| `e2e` | — | Files-only |
| `resource` | `resrc` | Creates full feature slice; import generated module when ready |

`fluo generate resource <name>` creates a module, controller, service, repository, request DTO, response DTO, and tests. It does not wire the resource module into a parent module automatically.

Use `--dry-run` to preview target resolution, skipped/overwritten files, module auto-registration, files-only wiring status, and next-step hints. `--force` turns changed existing-file plan entries from `SKIP` into `OVERWRITE`.

Generator discovery is intentionally limited to the built-in `@fluojs/cli/builtin` collection. The CLI does not load arbitrary workspace generator code.

## Diagnostics commands

Use these commands when debugging CLI installation, project scripts, runtime facts, or update state.

```bash
fluo doctor
fluo info
fluo analyze
```

`fluo analyze` is read-only and points to deeper `inspect --report` and `migrate --json` workflows.

Use package helpers for first-party packages and upgrades:

```bash
fluo add studio --dev --dry-run
fluo upgrade
```

## Decorator codemods

Use `fluo migrate` to align code with TC39 standard decorators.

```bash
fluo migrate ./src
fluo migrate ./src --json
fluo migrate ./src --apply
fluo migrate ./src --apply --json
fluo migrate ./src --only imports,inject-params
fluo migrate ./src --skip tests
```

Key transformations include:

- rewriting imports from `@nestjs/common` to `@fluojs/core` or `@fluojs/http`,
- rewriting bootstrap/listen patterns into fluo runtime startup conventions,
- migrating constructor parameter `@Inject(...)` usage into fluo-compatible dependency declarations,
- removing `@Injectable()` and mapping scopes to `@Scope()`,
- migrating supported test templates toward `@fluojs/testing`,
- updating `tsconfig.json` for standard decorators and TS6-safe paths.

JSON mode emits a stable machine-readable report and keeps parser errors or invalid flags on stderr with exit code `1`.

## Runtime inspection

Use `fluo inspect` to export module graph and diagnostics snapshots.

```bash
fluo inspect ./src/app.module.ts --json --output artifacts/inspect-snapshot.json
fluo inspect ./src/app.module.ts --json --timing
fluo inspect ./src/app.module.ts --report --output artifacts/inspect-report.json
fluo inspect ./src/app.module.ts --export AdminModule --json
```

Mermaid output is delegated to Studio:

```bash
pnpm add -D @fluojs/studio
fluo inspect ./src/app.module.ts --mermaid
```

The runtime produces the snapshot. The CLI serializes it, selects a named module export with `--export`, includes timing diagnostics with `--timing`, and writes explicit artifacts with `--output`.

## Public API summary

| Export | Description |
| --- | --- |
| `runCli(argv?, options?)` | Programmatic entrypoint for CLI commands. |
| `newUsage()` | Current `fluo new` usage text. |
| `runNewCommand(argv, options?)` | Programmatic scaffolding entrypoint. |
| `CliPromptCancelledError` | Stable sentinel for normal prompt cancellation. |
| `GenerateOptions`, `GeneratedFile`, `GeneratorKind`, `ModuleRegistration` | Programmatic generator types. |

Programmatic entrypoints return numeric exit codes instead of calling `process.exit(...)`. Dependency installation and git initialization run only when resolved `fluo new` options request them.
