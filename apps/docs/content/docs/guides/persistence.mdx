title: Persistence
description: Choose and operate database, Redis, cache, transaction, and queue packages in fluo applications.

Persistence in fluo is explicit by design. The framework does not hide every database behind one generic repository abstraction. Instead, each package keeps the semantics of its backing technology visible while connecting it to fluo's DI, lifecycle, interceptor, and status contracts.

Use this guide when you need durable records, shared Redis state, HTTP/application caching, or background job processing.

## Package map

| Need | Package | Primary contract |
| --- | --- | --- |
| Prisma ORM | `@fluojs/prisma` | `PrismaModule`, `PrismaService.current()`, request transactions |
| Drizzle ORM | `@fluojs/drizzle` | `DrizzleModule`, `DrizzleDatabase.current()`, optional dispose hook |
| Mongoose ODM | `@fluojs/mongoose` | `MongooseModule`, `MongooseConnection`, explicit sessions |
| Shared Redis clients | `@fluojs/redis` | `RedisModule`, `RedisService`, named clients |
| HTTP and app caching | `@fluojs/cache-manager` | `CacheModule`, `CacheInterceptor`, `CacheService` |
| Background jobs | `@fluojs/queue` | `QueueModule`, `QueueLifecycleService`, `@QueueWorker()` |

## Choosing a persistence package

| If you need... | Choose | Why |
| --- | --- | --- |
| Type-safe relational access with generated Prisma clients | `@fluojs/prisma` | Keeps generated Prisma types while adding lifecycle and ALS-backed transaction context. |
| Drizzle query builder integration | `@fluojs/drizzle` | Wraps a Drizzle database handle with `current()`, request transactions, and shutdown status. |
| MongoDB with Mongoose models and sessions | `@fluojs/mongoose` | Preserves explicit Mongoose session semantics instead of rewriting operation options. |
| Shared key-value state for cache, queues, throttling, pub/sub | `@fluojs/redis` | Owns Redis client bootstrap/shutdown and named client tokens. |
| Cached GET responses or `remember(...)` style application caching | `@fluojs/cache-manager` | Offers memory, Redis, and custom store contracts with bounded ownership. |
| Retryable work outside the request path | `@fluojs/queue` | Uses Redis-backed distributed jobs, worker discovery, retry/backoff, and dead-letter records. |

## Prisma

Register a `PrismaClient` through `PrismaModule.forRoot(...)`.

```ts
import { Module } from '@fluojs/core';
import { PrismaModule } from '@fluojs/prisma';
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

@Module({
  imports: [PrismaModule.forRoot({ client: prisma })],
})
class AppModule {}
```

Use `PrismaService.current()` inside repositories. It returns the active transaction client when a transaction context exists, otherwise the root client.

```ts
import { Inject } from '@fluojs/core';
import { PrismaService } from '@fluojs/prisma';
import { PrismaClient } from '@prisma/client';

@Inject(PrismaService)
export class UserRepository {
  constructor(private readonly prisma: PrismaService<PrismaClient>) {}

  async findById(id: string) {
    return this.prisma.current().user.findUnique({ where: { id } });
  }
}
```

For manual transactions, use `transaction(...)` and keep repository code on `current()`.

```ts
await this.prisma.transaction(async () => {
  const user = await this.prisma.current().user.create({ data });
  await this.prisma.current().profile.create({ data: { userId: user.id } });
});
```

For request-scoped transactions on the default unnamed client, apply `PrismaTransactionInterceptor`.

```ts
import { Post, UseInterceptors } from '@fluojs/http';
import { PrismaTransactionInterceptor } from '@fluojs/prisma';

@UseInterceptors(PrismaTransactionInterceptor)
class UserController {
  @Post()
  async create() {
    // Downstream PrismaService.current() calls share this transaction.
  }
}
```

Named clients are supported with `PrismaModule.forRoot({ name, client })` and `getPrismaServiceToken(name)`. Use names for every additional client so token resolution stays explicit.

## Drizzle

Drizzle registration usually comes from an injected config provider because the underlying driver owns its connection pool.

```ts
import { ConfigService } from '@fluojs/config';
import { Module } from '@fluojs/core';
import { DrizzleModule } from '@fluojs/drizzle';
import { drizzle } from 'drizzle-orm/node-postgres';
import { Pool } from 'pg';

@Module({
  imports: [
    DrizzleModule.forRootAsync({
      inject: [ConfigService],
      useFactory: async (config: ConfigService) => {
        const pool = new Pool({ connectionString: config.getOrThrow<string>('DATABASE_URL') });

        return {
          database: drizzle(pool),
          dispose: async () => {
            await pool.end();
          },
        };
      },
    }),
  ],
})
export class AppModule {}
```

Repository code uses `DrizzleDatabase.current()` so it automatically sees the active transaction handle.

```ts
import { DrizzleDatabase } from '@fluojs/drizzle';
import { eq } from 'drizzle-orm';
import { users } from './schema';

export class UserRepository {
  constructor(private readonly db: DrizzleDatabase) {}

  async findById(id: string) {
    return this.db.current().select().from(users).where(eq(users.id, id));
  }
}
```

`DrizzleTransactionInterceptor` wraps HTTP requests with `DrizzleDatabase.requestTransaction(...)`. During shutdown, active request transactions are aborted and settled before the configured `dispose(database)` hook runs.

## Mongoose

`@fluojs/mongoose` plugs an existing Mongoose connection into fluo lifecycle management. The package does not create or own the raw connection unless your `dispose` hook does that work.

```ts
import { Module } from '@fluojs/core';
import { MongooseModule } from '@fluojs/mongoose';
import mongoose from 'mongoose';

const connection = mongoose.createConnection('mongodb://localhost:27017/test');

@Module({
  imports: [
    MongooseModule.forRoot({
      connection,
      dispose: async (conn) => conn.close(),
    }),
  ],
})
class AppModule {}
```

Use `MongooseConnection.current()` to access the connection and `currentSession()` inside transactions.

```ts
import { MongooseConnection } from '@fluojs/mongoose';

export class UserRepository {
  constructor(private readonly conn: MongooseConnection) {}

  async findById(id: string) {
    const User = this.conn.current().model('User');
    return User.findById(id);
  }
}
```

Mongoose transactions still require explicit session options on model calls.

```ts
await this.conn.transaction(async () => {
  const session = this.conn.currentSession();
  const User = this.conn.current().model('User');

  await User.create([{ name: 'Ada' }], { session });
});
```

fluo never rewrites Mongoose operation options. If a model call omits `{ session }`, repositories should not assume fluo will attach one automatically.

## Redis

Use Redis when multiple packages need shared ephemeral state: cache storage, queues, throttling, fan-out, or custom key-value operations.

```ts
import { Module } from '@fluojs/core';
import { RedisModule } from '@fluojs/redis';

@Module({
  imports: [
    RedisModule.forRoot({
      host: 'localhost',
      port: 6379,
    }),
  ],
})
export class AppModule {}
```

Inject `RedisService` for JSON-aware `get`/`set`/`del` operations, or inject `REDIS_CLIENT` for the raw `ioredis` instance.

```ts
import { Inject } from '@fluojs/core';
import { RedisService } from '@fluojs/redis';

@Inject(RedisService)
export class CacheRepository {
  constructor(private readonly redis: RedisService) {}

  async saveUser(id: string, user: object) {
    await this.redis.set(`user:${id}`, user, 3600);
  }

  async getUser(id: string) {
    return await this.redis.get(`user:${id}`);
  }
}
```

Redis clients created by `RedisModule` are lifecycle-managed. fluo forces `lazyConnect: true`, opens sockets during bootstrap, attempts graceful `quit()` during shutdown, and falls back to `disconnect()` when necessary.

Use named clients with `RedisModule.forRoot({ name, ...options })`, `getRedisClientToken(name)`, and `getRedisServiceToken(name)` when one app needs more than one Redis connection.

## Cache manager

`@fluojs/cache-manager` supports HTTP response caching and programmatic application caching.

```ts
import { Module } from '@fluojs/core';
import { Controller, Get, UseInterceptors } from '@fluojs/http';
import { CacheInterceptor, CacheModule, CacheTTL } from '@fluojs/cache-manager';

@Controller('/products')
class ProductController {
  @Get('/')
  @UseInterceptors(CacheInterceptor)
  @CacheTTL(60)
  list() {
    return [{ id: 1, name: 'Product A' }];
  }
}

@Module({
  imports: [CacheModule.forRoot({ store: 'memory' })],
  controllers: [ProductController],
})
class AppModule {}
```

For application-level caching, inject `CacheService`.

```ts
import { Inject } from '@fluojs/core';
import { CacheService } from '@fluojs/cache-manager';

class UserService {
  constructor(@Inject(CacheService) private readonly cache: CacheService) {}

  async getProfile(userId: string) {
    return this.cache.remember(`user:${userId}`, async () => fetchUserProfile(userId), 300);
  }
}
```

Important cache semantics:

- Memory cache defaults to a 300-second TTL and at most 1,000 live entries when `ttl` is omitted.
- Redis cache stores entries with `JSON.stringify(...)`; normalize `Date`, `bigint`, functions, symbols, and cyclic graphs before caching.
- `CacheService.reset()` clears only entries owned by the configured store or Redis `keyPrefix`.
- HTTP caching stores successful, uncommitted GET results only. It skips streams, committed responses, `undefined`, and non-2xx responses.

## Queue

Use queues when work should move out of the request path and survive process boundaries.

```ts
import { QueueWorker } from '@fluojs/queue';

export class ProcessOrderJob {
  constructor(public readonly orderId: string) {}
}

@QueueWorker(ProcessOrderJob, { attempts: 3, backoff: { type: 'fixed', delayMs: 5000 } })
export class OrderWorker {
  async handle(job: ProcessOrderJob) {
    console.log(`Processing order: ${job.orderId}`);
  }
}
```

Register Redis and `QueueModule`, then enqueue jobs through `QueueLifecycleService`.

```ts
import { Inject, Module } from '@fluojs/core';
import { QueueLifecycleService, QueueModule } from '@fluojs/queue';
import { RedisModule } from '@fluojs/redis';

@Module({
  imports: [
    RedisModule.forRoot({ host: 'localhost', port: 6379 }),
    QueueModule.forRoot(),
  ],
  providers: [OrderWorker],
})
export class AppModule {}

export class OrderService {
  @Inject(QueueLifecycleService)
  private readonly queue: QueueLifecycleService;

  async placeOrder(id: string) {
    await this.queue.enqueue(new ProcessOrderJob(id));
  }
}
```

Queue jobs must be JSON-serializable plain objects. Queue serializes the payload before enqueueing and rehydrates the job prototype on the worker side. When retries are exhausted, Queue appends a dead-letter record to Redis under `fluo:queue:dead-letter:<jobName>` for manual inspection or recovery.

## Transaction and lifecycle rules

| Topic | Rule |
| --- | --- |
| Repository access | Use the package's `current()` seam (`PrismaService.current()`, `DrizzleDatabase.current()`, `MongooseConnection.current()`). |
| Request transactions | Use interceptors for default HTTP request boundaries; named/multiple clients often need explicit transaction calls. |
| Nested transactions | Active boundaries are reused; nested options that would be ignored are rejected. |
| Shutdown | Packages drain or abort active request transactions before disposing underlying resources. |
| Strict transactions | Enable strict mode when direct execution fallback would hide a missing driver capability. |
| Redis ownership | `RedisModule` owns clients it creates; consuming packages own only duplicated or package-created resources. |
| Cache reset | Reset cache-owned namespaces only; never share cache prefixes with unrelated application data. |
| Queue workers | Only singleton `@QueueWorker()` providers/controllers are registered. |

## Common mistakes

| Mistake | Better approach |
| --- | --- |
| Passing `tx` objects through every service manually | Put repository access behind `current()` and open explicit transaction boundaries. |
| Assuming Mongoose sessions are attached automatically | Pass `currentSession()` into Mongoose operations that participate in a transaction. |
| Using memory cache or in-memory throttling as shared production state | Use Redis-backed storage for multi-instance deployments. |
| Caching non-JSON values in Redis | Normalize values before caching. |
| Sharing Redis cache prefixes with application data | Use a dedicated `keyPrefix`, for example `myapp:cache:`. |
| Enqueueing class instances with non-serializable payloads | Keep job payloads JSON-serializable and reconstruct rich objects in the worker. |
