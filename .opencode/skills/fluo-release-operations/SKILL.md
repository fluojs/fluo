# Fluo Release Operations

This skill provides the knowledge required for managing releases within the fluo repository. It enforces the "Changesets-only" workflow and prevents unsafe local publishing.

## Core Policies

1. **Changesets ONLY**: The sole source of truth for versioning and changelogs.
2. **No Local Publish**: `npm publish` must NEVER be run locally. All publishing is handled via GitHub Actions.
3. **GitHub Actions Authority**: The canonical release workflow is located at `.github/workflows/release.yml`.

## Release Lane Policy

- **`main` is patch-only**: Treat `main` as the stable patch lane. Pull requests targeting `main` must not introduce `minor` or `major` changesets.
- **Hold minor/major separately**: Keep `minor` and `major` release intents on dedicated release or prerelease branches until maintainers intentionally promote them.
- **Check generated Version Packages PRs**: Before approving or merging a Version Packages PR, verify both its base branch and generated `packages/*/package.json` version deltas. A Version Packages PR targeting `main` must contain only patch bumps.
- **Patch release sequence**: Merge patch-only implementation PRs to `main`, allow the Changesets action to create a patch-only Version Packages PR, then merge that Version Packages PR to publish the patch release through GitHub Actions.
- **Major/minor release sequence**: Use a dedicated release/prerelease branch for non-patch release trains. Do not retarget a minor/major Version Packages PR to `main` unless its generated version deltas are intentionally approved for stable release.

## Release Lane Verification

- Run `pnpm verify:changeset-release-lane -- --lane=stable --base-ref=<ref>` when checking whether a branch is safe for the `main` patch lane.
- The lane verifier rejects pending `.changeset/*.md` files with `minor` or `major` bumps and generated package version deltas that would publish minor/major versions on the stable lane.
- A failing stable lane check is not a reason to downgrade semver. Move the release intent and its implementation to the correct release/prerelease lane instead.

## Release Metadata Sources

- **Changesets**: `.changeset/*.md` files define pending releases.
- **Config**: `.changeset/config.json` contains the workspace release configuration.
- **Changelogs**: Package-level `CHANGELOG.md` files are generated by the release process.
- **Governance**: `docs/contracts/release-governance.md` defines the overarching release policy.

## Workflow knowledge

- **Version Packages PR**: A PR automatically generated by the Changesets bot that consumes pending `.changeset/*.md` files and updates package versions.
- **Patch Version Packages PR**: A Version Packages PR targeting `main`; it must be patch-only and pass the stable release lane verifier.
- **Minor/Major Version Packages PR**: A Version Packages PR targeting a dedicated release/prerelease branch; keep it away from `main` until the train is intentionally promoted.
- **Publish Workflow**: Triggered after the "Version Packages" PR is merged to `main`; because `main` is patch-only, this path is for stable patch publication unless maintainers explicitly change the lane policy.
- **Release Intents**: `tooling/release/intents/*.json` are legacy references and should not be used as the primary source of truth for new work.

## Checklist for Changeset Creation

- [ ] Identify all affected public `@fluojs/*` packages.
- [ ] Determine the semver bump level (`patch`, `minor`, `major`) based on the behavioral contract impact.
- [ ] Provide a clear, human-readable summary of the change in the changeset file.
- [ ] Ensure the changeset is committed alongside the functional changes.
